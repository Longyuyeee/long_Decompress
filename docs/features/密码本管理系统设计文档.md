# 胧解压·方便助手 - 密码本管理系统设计文档

## 1. 系统概述

### 1.1 设计目标
为"胧解压·方便助手"提供安全、智能、易用的密码管理功能，支持压缩文件密码的存储、检索、生成和智能匹配。

### 1.2 核心功能

- 安全加密机制

## 2. 密码本存储格式设计

### 2.1 方案对比

#### 2.1.1 文本文件方案
**优点：**
- 简单易用，无需额外依赖
- 人类可读，便于手动编辑
- 跨平台兼容性好
- 易于版本控制

**缺点：**
- 安全性差（明文存储）
- 查询效率低
- 缺乏结构化数据支持
- 扩展性有限

**建议格式：**
```
# 密码本格式
文件名: 密码
压缩包1.zip: password123
压缩包2.rar: mySecret@2024
# 标签: 游戏, 工作
压缩包3.7z: gamePass!2024
```

#### 2.1.2 JSON文件方案
**优点：**
- 结构化数据，便于程序处理
- 支持复杂数据类型
- 易于扩展和修改
- 良好的可读性

**缺点：**
- 安全性需要额外加密
- 文件较大时解析效率低
- 手动编辑容易出错

**建议格式：**
```json
{
  "version": "1.0",
  "passwords": [
    {
      "id": "uuid-1234",
      "filename": "压缩包1.zip",
      "password": "encrypted_data",
      "tags": ["游戏", "常用"],
      "strength": "medium",
      "usage_count": 15,
      "last_used": "2024-03-08T10:30:00Z",
      "created_at": "2024-01-01T00:00:00Z",
      "source": "manual"
    }
  ],
  "metadata": {
    "total_count": 50,
    "last_modified": "2024-03-08T10:30:00Z",
    "encryption_method": "AES-256-GCM"
  }
}
```

#### 2.1.3 SQLite数据库方案
**优点：**
- 高效的查询性能
- 事务支持，数据一致性
- 支持复杂查询和索引
- 良好的扩展性
- 内置加密支持（SQLCipher）

**缺点：**
- 需要SQLite依赖
- 文件格式不透明
- 手动编辑困难
- 相对复杂

**建议数据库结构：**
```sql
-- 密码表
CREATE TABLE passwords (
    id TEXT PRIMARY KEY,
    filename TEXT NOT NULL,
    encrypted_password TEXT NOT NULL,
    file_hash TEXT,  -- 文件哈希用于精确匹配
    tags TEXT,  -- JSON数组存储
    category TEXT,
    strength INTEGER,  -- 1-5级强度
    usage_count INTEGER DEFAULT 0,
    last_used TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    source TEXT,
    notes TEXT
);

-- 标签表
CREATE TABLE tags (
    id INTEGER PRIMARY KEY,
    name TEXT UNIQUE NOT NULL,
    color TEXT,
    description TEXT
);

-- 密码-标签关联表
CREATE TABLE password_tags (
    password_id TEXT,
    tag_id INTEGER,
    PRIMARY KEY (password_id, tag_id),
    FOREIGN KEY (password_id) REFERENCES passwords(id),
    FOREIGN KEY (tag_id) REFERENCES tags(id)
);

-- 使用历史表
CREATE TABLE usage_history (
    id INTEGER PRIMARY KEY,
    password_id TEXT,
    filename TEXT,
    success BOOLEAN,
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (password_id) REFERENCES passwords(id)
);
```

### 2.2 推荐方案：SQLite + SQLCipher
**理由：**
1. 安全性：SQLCipher提供透明的AES-256加密
2. 性能：高效的索引和查询
3. 扩展性：易于添加新功能
4. 可靠性：事务支持保证数据完整性
5. 跨平台：SQLite广泛支持

def calculate_priority(password_record):
    # 使用频率权重：40%
    frequency_score = normalize(log(usage_count + 1)) * 0.4

    # 时间衰减权重：30%
    recency_score = time_decay(last_used) * 0.3

    # 匹配度权重：20%
    similarity_score = filename_similarity(current_file, password_record.filename) * 0.2

    # 成功率权重：10%
    success_rate = success_count / max(usage_count, 1) * 0.1

#### 5.1.2 时间衰减函数
```python
def time_decay(last_used_timestamp, half_life_days=30):
    # 指数衰减，半衰期30天
    days_passed = (current_time - last_used_timestamp).days
    return 0.5 ** (days_passed / half_life_days)
```

```python
def calculate_similarity(str1, str2):
    # 1. 编辑距离相似度
    edit_distance = levenshtein_distance(str1, str2)
    max_len = max(len(str1), len(str2))
    edit_similarity = 1 - (edit_distance / max_len)

    # 2. 公共子序列相似度
    lcs_len = longest_common_subsequence(str1, str2)
    lcs_similarity = (2 * lcs_len) / (len(str1) + len(str2))

    # 3. 关键词重叠相似度
    keywords1 = extract_keywords(str1)
    keywords2 = extract_keywords(str2)
    overlap = len(set(keywords1) & set(keywords2))
    keyword_similarity = overlap / max(len(keywords1), len(keywords2), 1)

    # 加权组合
    return (edit_similarity * 0.4 +
            lcs_similarity * 0.3 +
            keyword_similarity * 0.3)
```
