## 1. 沟通规范

### 1.1 会议安排
| 会议类型 | 频率 | 时间 | 时长 | 参与人员 | 目的 |
|----------|------|------|------|----------|------|
| 每日站会 | 每日 | 9:30 | 15分钟 | 全体成员 | 同步进展、计划、问题 |

- **即时通讯**: 使用团队聊天工具（如Slack、钉钉）
- **文档协作**: 使用Markdown + Git进行文档管理
- **任务跟踪**: 使用任务管理系统
- **代码审查**: 使用Git Pull Request流程

### 1.3 沟通礼仪
1. **及时响应**: 工作时间内2小时内响应
2. **明确主题**: 消息标题清晰，内容简洁
3. **尊重时间**: 会议准时开始和结束
4. **积极倾听**: 尊重他人意见，充分讨论

## 2. 代码协作规范

### 2.1 分支管理策略
```
main
  ├── develop (开发分支)
  │   ├── feature/fe-001-design-system (前端特性分支)
  │   ├── feature/be-001-database (后端特性分支)
  │   ├── feature/pw-001-password-db (密码本特性分支)
  │   └── hotfix/紧急修复分支
  └── release/v1.0.0 (发布分支)
```

### 2.2 分支命名规范
- **特性分支**: `feature/{任务ID}-{简短描述}`
  - 示例: `feature/fe-001-design-system`
- **修复分支**: `hotfix/{问题描述}`
  - 示例: `hotfix/fix-memory-leak`
- **发布分支**: `release/v{版本号}`
  - 示例: `release/v1.0.0`

### 2.3 提交信息规范
使用约定式提交格式：
```
<类型>(<范围>): <描述>

[可选正文]

[可选脚注]
```

**类型**:
- `feat`: 新功能
- `fix`: 修复bug
- `docs`: 文档更新
- `style`: 代码格式调整
- `refactor`: 代码重构
- `test`: 测试相关
- `chore`: 构建过程或辅助工具变动

**示例**:
```
feat(compression): 添加ZIP解压功能

- 实现基础ZIP解压逻辑
- 添加错误处理机制
- 更新相关测试用例

Closes #BE-004
```

### 2.4 Pull Request流程
1. **创建PR**: 从特性分支向develop分支创建PR
2. **填写模板**: 使用PR模板，描述变更内容
3. **代码审查**: 至少需要1人审查通过
4. **CI检查**: 所有CI检查必须通过
5. **合并**: 使用Squash Merge，保持提交历史整洁

3. 定期提交代码

1. 完成所有开发工作
2. 通过所有测试

/// 解压ZIP文件
///
/// # 参数
/// - `file_path`: 文件路径 (String)
/// - `output_dir`: 输出目录 (Option<String>)
/// - `password`: 可选密码 (Option<String>)
///
/// # 返回值
/// 返回解压结果路径 (String) 或错误信息
///
/// # 错误
/// - 文件不存在: "File not found"
/// - 密码错误: "Invalid password"
/// - 格式不支持: "Unsupported format"
///
/// # 示例
/// ```typescript
/// invoke('extract_file', {
///   filePath: '/path/to/file.zip',
///   outputDir: '/output/dir',
///   password: 'secret'
/// })
/// ```
#[command]
pub async fn extract_file(
    file_path: String,
    output_dir: Option<String>,
    password: Option<String>,
) -> Result<String, String> {
    match CompressionService::extract(&file_path, output_dir.as_deref(), password.as_deref()).await {
        Ok(output) => Ok(output),
        Err(e) => Err(format!("解压失败: {}", e)),
    }
}

```rust
///
/// # 参数
/// - `file_path`: 文件路径

/// - `password`: 可选密码

/// # 返回值
/// 返回解压结果路径

/// # 错误
/// 可能返回文件不存在、密码错误等错误
pub async fn extract_zip(
    zip_path: &Path,
    output_dir: &Path,
    password: Option<&str>,
) -> Result<()> {
    let file = File::open(zip_path).context("打开ZIP文件失败")?;
    let mut archive = ZipArchive::new(file).context("读取ZIP归档失败")?;

    for i in 0..archive.len() {
        let mut file = archive.by_index(i).context("获取ZIP文件条目失败")?;

        // 处理加密文件
        if file.is_encrypted() {
            if let Some(pwd) = password {
                if file.set_password(pwd) {
                    // 密码设置成功
                } else {
                    return Err(anyhow::anyhow!("ZIP文件需要密码"));
                }
            } else {
                return Err(anyhow::anyhow!("ZIP文件需要密码"));
            }
        }

        // ... 文件解压逻辑
    }

    Ok(())
}
```

```vue
<script setup lang="ts">
import { ref, computed, onMounted } from 'vue'
import { useFileStore } from '@/stores/file'
import { invoke } from '@tauri-apps/api/core'

// 类型定义
interface Props {
  /** 组件标题 */
  title: string
  /** 是否禁用 */
  disabled?: boolean
  /** 文件类型过滤 */
  accept?: string
}

interface Emits {
  /** 文件选择事件 */
  (e: 'select', files: File[]): void
  /** 错误事件 */
  (e: 'error', message: string): void
}

// Props和Emits定义
const props = withDefaults(defineProps<Props>(), {
  disabled: false,
  accept: '*/*'
})

const emit = defineEmits<Emits>()

// 状态管理
const fileStore = useFileStore()
const isLoading = ref(false)
const selectedFiles = ref<File[]>([])

// 计算属性
const fileCount = computed(() => selectedFiles.value.length)
const totalSize = computed(() =>
  selectedFiles.value.reduce((sum, file) => sum + file.size, 0)
)

// 方法
const handleFileSelect = async (event: Event) => {
  const input = event.target as HTMLInputElement
  if (!input.files?.length) return

  const files = Array.from(input.files)
  selectedFiles.value = files
  emit('select', files)

  // 调用Tauri后端API
  try {
    isLoading.value = true
    const filePaths = files.map(file => file.path)
    const result = await invoke('process_files', { files: filePaths })
    console.log('处理结果:', result)
  } catch (error) {
    emit('error', error instanceof Error ? error.message : '处理失败')
  } finally {
    isLoading.value = false
  }
}

const clearSelection = () => {
  selectedFiles.value = []
}

// 生命周期
onMounted(() => {
  console.log('FileSelector组件已挂载')
})
</script>
```

## 6. 问题解决流程

### 6.1 Bug报告流程
1. **发现bug**: 在测试或使用过程中发现bug
2. **创建issue**: 使用issue模板创建bug报告
3. **分配处理**: 项目管理领队分配处理人员
4. **修复验证**: 修复后需要原报告人验证
5. **关闭issue**: 验证通过后关闭issue

### 6.2 Issue模板
```markdown
## 问题描述
{清晰描述问题现象}

## 重现步骤
1. {步骤1}
2. {步骤2}
3. {步骤3}

## 预期行为
{期望的正确行为}

## 实际行为
{实际发生的错误行为}

## 环境信息
- 操作系统: {Windows 11}
- 应用版本: {v1.0.0}
- 其他相关信息

## 截图/日志
{相关截图或错误日志}

## 严重程度
- [ ] 致命（系统崩溃）
- [ ] 严重（功能不可用）
- [ ] 一般（功能受限）

```

### 6.3 技术问题解决
1. **个人尝试**: 首先尝试自己解决问题（30分钟）
2. **团队求助**: 在团队群中提问（附上详细上下文）
3. **技术讨论**: 安排技术讨论会议
4. **外部求助**: 查阅文档、搜索解决方案

## 7. 质量保证

### 7.1 代码审查要点
- **功能正确性**: 代码是否实现需求
- **代码质量**: 是否符合编码规范
- **测试覆盖**: 是否有足够的测试
- **安全性**: 是否存在安全漏洞
- **性能**: 是否有性能问题

### 7.2 审查意见格式
```
## 总体评价
{总体评价}

## 具体意见

- 第10行: {具体意见}
- 第25行: {具体意见}

- {改进建议1}
```

### 7.3 审查响应
- **接受意见**: 修改代码并回复
- **讨论意见**: 提出不同看法，进行讨论
- **拒绝意见**: 说明拒绝理由（需谨慎）

## 8. 持续集成

### 8.1 CI流程
1. **代码检查**: rustfmt, clippy, eslint
2. **编译测试**: 所有平台编译测试
3. **单元测试**: 运行所有单元测试
4. **集成测试**: 运行集成测试
5. **构建产物**: 生成可执行文件

### 8.2 CI失败处理
1. **立即修复**: CI失败必须立即修复
2. **优先处理**: CI失败优先级高于新功能开发
3. **根本原因**: 分析失败原因，防止再次发生
