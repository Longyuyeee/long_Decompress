# 胧压缩·方便助手 - 性能测试套件设计文档

## 概述

本文档定义了性能测试套件的设计、实施计划和验收标准。性能测试套件旨在验证性能优化效果，确保系统在各种负载下的稳定性和性能表现。

## 测试目标

### 主要目标
1. **验证性能优化效果**：对比优化前后的性能指标
2. **确保系统稳定性**：在高负载和长时间运行下的稳定性
3. **识别性能瓶颈**：发现系统中的性能瓶颈
4. **建立性能基准**：为后续优化提供基准数据

### 具体指标
- **解压速度**：大文件（1GB）解压时间
- **并发处理能力**：同时处理多个压缩任务
- **内存使用效率**：长时间运行的内存稳定性
- **CPU利用率**：多核CPU的利用效率
- **磁盘IO效率**：文件读写性能

## 测试架构

### 测试金字塔
```
        E2E性能测试 (10%)
           ↑
    集成性能测试 (20%)
           ↑
    单元性能测试 (70%)
```

### 测试组件
1. **Rust基准测试** (`src-tauri/benches/`) - 使用Criterion库
2. **前端性能测试** (`tests/performance/`) - 使用Vitest和真实API调用
3. **集成性能测试** (`tests/integration/`) - 测试完整工作流程
4. **E2E性能测试** (`tests/e2e/`) - 使用Playwright测试真实用户场景

## 测试用例设计

### 1. 大文件解压测试 (1GB ZIP文件)

#### 测试场景
- 解压1GB的ZIP文件，包含多种文件类型
- 对比优化前后的解压速度
- 监控CPU和内存使用情况

#### 测试数据
- 生成1GB测试ZIP文件，包含：
  - 50%文本文件（可压缩）
  - 30%二进制文件（中等压缩）
  - 20%已压缩文件（低压缩）

#### 性能指标
| 指标 | 优化前目标 | 优化后目标 | 测量方法 |
|------|-----------|-----------|----------|
| 解压时间 | 基准 | ≤基准的50% | 计时器 |
| CPU使用率 | 基准 | 提高30% | 系统监控 |
| 内存峰值 | 基准 | ≤基准的80% | 内存监控 |
| 磁盘IO | 基准 | 提高20% | IO监控 |

### 2. 并发任务测试

#### 测试场景
- 同时解压多个ZIP文件（5个并发任务）
- 测试任务队列和调度性能
- 验证资源管理和错误处理

#### 测试数据
- 5个100MB ZIP文件，每个包含不同内容
- 同时提交解压任务

#### 性能指标
| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| 总完成时间 | ≤单个任务时间×2 | 计时器 |
| 任务调度延迟 | ≤100ms | 时间戳记录 |
| 资源争用 | 无死锁/饥饿 | 监控工具 |
| 错误处理 | 优雅降级 | 错误日志 |

### 3. 内存使用测试

#### 测试场景
- 长时间运行压力测试（30分钟）
- 批量处理大量小文件（1000个1MB文件）
- 监控内存泄漏和碎片化

#### 测试数据
- 1000个1MB文本文件
- 持续30分钟的压缩/解压循环

#### 性能指标

| GC频率 | 稳定 | 垃圾回收监控 |
| 响应时间 | 保持稳定 | 性能监控 |

## 测试工具和基础设施

### 1. 测试数据生成工具

#### 扩展现有工具
```bash
# 生成1GB测试ZIP文件
./scripts/generate_large_test_files.sh --size 1GB --type mixed

# 生成并发测试文件
./scripts/generate_concurrent_test_files.sh --count 5 --size 100MB

# 生成内存测试文件
./scripts/generate_memory_test_files.sh --count 1000 --size 1MB
```

#### 新工具需求
- **大文件生成器**：支持生成1GB+的测试文件
- **ZIP文件生成器**：创建包含特定内容的ZIP文件
- **性能数据收集器**：收集和聚合性能指标

### 2. 监控工具

#### 系统级监控
- **CPU使用率**：使用`psutil`或系统API
- **内存使用**：监控堆内存和系统内存
- **磁盘IO**：监控读写速度和延迟
- **网络IO**：监控API调用性能

#### 应用级监控
- **函数执行时间**：使用性能分析器
- **内存分配**：使用内存分析工具
- **并发控制**：监控锁竞争和线程状态

### 3. 测试框架

#### Rust基准测试
```rust
// 使用Criterion库进行基准测试
criterion_group! {
    name = benches;
    config = Criterion::default()
        .sample_size(10)
        .warm_up_time(Duration::from_secs(3))
        .measurement_time(Duration::from_secs(10));
    targets = bench_large_file_extraction, bench_concurrent_extraction
}
```

#### 前端性能测试
```typescript
// 使用Vitest进行性能测试
describe('性能测试', () => {
  it('应该在大文件解压时保持性能', async () => {
    const startTime = performance.now()
    const result = await compressionService.extractLargeFile()
    const endTime = performance.now()

    expect(endTime - startTime).toBeLessThan(10000) // 10秒内完成
  })
})
```

## 风险评估和缓解

### 技术风险
1. **测试数据过大**
   - 风险：1GB测试文件可能占用大量磁盘空间
   - 缓解：使用临时文件，测试后自动清理
   - 状态：已考虑

2. **测试时间过长**
   - 风险：长时间测试可能影响开发流程
   - 缓解：在CI/CD中设置超时，使用并行测试
   - 状态：已考虑

3. **环境差异**
   - 风险：不同环境性能表现可能不同
   - 缓解：建立基线，使用相对比较
   - 状态：已考虑

### 资源风险
1. **硬件要求**
   - 风险：性能测试需要足够资源
   - 缓解：使用专用测试环境
   - 状态：需要协调

2. **时间投入**
   - 风险：测试开发需要时间
   - 缓解：分阶段实施，优先关键测试
   - 状态：已规划
