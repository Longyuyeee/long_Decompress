# 解压全流程与智能干预系统设计文档 (Extraction Full-Flow & Intervention Design)

## 1. 系统概述 (Overview)
本系统旨在打造一个“高度自动化但关键节点可控”的现代化解压中心。系统默认以静默方式处理所有任务（包括密码尝试），仅在遇到不可逾越的障碍（如文件冲突）或需要耗费大量资源（算法破解）时，才会挂起任务并向用户请求指令。

## 2. 核心状态机与并发模型

### 2.1 任务生命周期 (Task Lifecycle)
1. **Pending (排队中)**：任务进入队列，等待空闲的并发 Slot。
2. **Pre-checking (预检与静默解密)**：
   - 嗅探格式。
   - **常规密码本**：从数据库按频次提取高频密码尝试。
   - **智能猜测**：基于文件名/路径/日期生成密码尝试。
   - **暴力字典**：(如果设置中开启) 流式读取用户配置的 JSON 字典尝试。
3. **AlgorithmRequired (挂起：需算法破解)**：
   - 如果上述所有字典流均失效（且用户开启了暴力破解选项），任务进入此挂起态。
   - **关键特征**：任务在前端显示“需要算法破解”按钮。
4. **Extracting (解压中)**：正常进行文件数据拷贝。
5. **Conflict (挂起：文件冲突)**：
   - 遇到同名文件，触发防覆盖机制。
   - **关键特征**：任务挂起，前端显示“解决冲突”操作组。
6. **Completed / Failed (终态)**：
   - **Completed**：成功解压，显示“打开文件夹”。
   - **Failed**：由于各种原因（损坏、用户取消、无密码等）失败。**失败的任务不会从列表中消失**，而是保留在界面上以供用户查看错误日志。

### 2.2 并发与 Slot 占用原则
- **全局并发上限**：由用户在设置中配置（例如 3 个并发）。
- **挂起即占用**：当一个任务处于 `Conflict` 或 `AlgorithmRequired` 的**挂起态 (Paused)** 时，**它依然占用一个并发 Slot**。这防止了底层 IO 句柄的混乱转移，同时强制用户处理积压的问题。
- **自动顺延**：当一个任务到达终态（`Completed` 或 `Failed`）时，释放 Slot，队列自动拉取下一个 `Pending` 任务开始执行。

## 3. 任务分解 (Work Items) - 后端架构 (Backend)

### [BE-FLOW-001] 解压任务挂起与恢复机制 (Suspension Engine)
- **目标**：在不终止底层解压线程的前提下，让任务停在原地等待用户指令，且该任务依然持有 `Semaphore` 许可。
- **方案**：
    - **状态通道**：为每个被调度的任务创建一个或多个 `tokio::sync::oneshot::channel`。
    - **冲突挂起**：当检测到目标路径存在同名文件时，后端发送 `file-conflict` 事件给前端，并立即在一个 `.await` 处阻塞（等待 `oneshot::Receiver` 的信号）。
    - **算法授权挂起**：当字典手段用尽，后端发送 `algorithm-required` 事件给前端，同样挂起。
    - **恢复指令**：实现 Tauri Command（如 `resolve_conflict(task_id, strategy)` 和 `authorize_algorithm(task_id, proceed)`）。这些 Command 会查找到对应的 `oneshot::Sender` 并发送信号，从而唤醒挂起的任务继续执行。

### [BE-PASS-003] JSON 暴力字典流式加载器 (Brute-force JSON Loader)
- **目标**：支持处理包含百万级密码的外部 JSON 字典文件，防止内存溢出。
- **方案**：
    - 在 `PasswordAttemptService` 中实现 `attempt_from_json_dict`。
    - 使用 `tokio::io::BufReader` 和 `AsyncBufReadExt::lines()` 按行流式读取用户配置的 JSON/TXT 字典。
    - 在循环内部调用已有的 `test_archive_password`，并在每次尝试之间插入极短的 `tokio::task::yield_now()` 以防止阻塞其他异步任务。

### [BE-PASS-004] 密码尝试全链路集成 (The Attempt Chain)
- **目标**：将所有密码获取策略按正确的权重串联起来。
- **方案重构 `attempt_passwords_smartly`**：
    1. **Level 1 (高频常规)**：读取数据库 Top N 密码。
    2. **Level 2 (智能上下文)**：调用 `guess_passwords_from_context` 尝试文件名衍生密码。
    3. **Level 3 (暴力字典)**：检查 ConfigStore，若开启则启动 `BE-PASS-003` 的 JSON 流式字典。
    4. **Level 4 (挂起拦截)**：如果上述全军覆没，且允许算法破解，发送 `AlgorithmRequired` 事件并挂起。若不允许或未授权，则标记失败（保留记录但不丢弃）。

## 4. 任务分解 (Work Items) - 前端交互 (Frontend)

### [FE-FLOW-001] 拦截器视图与交互流 (Interception UI & Flow)
- **目标**：以非侵入式方式引导用户处理挂起任务。
- **任务卡片表现 (Card View)**：
    - 当任务进入 `Conflict` 或 `AlgorithmRequired` 状态时，卡片的状态提示区更新为对应的警告文案（如：“发现同名文件冲突” 或 “常规密码匹配失败”）。
    - 在状态文案旁显示一个显著的 **【去解决】** 按钮。
- **决策弹窗逻辑 (Resolution Dialogs)**：
    - **文件冲突弹窗**：
        - **详细对比**：展示来源文件与目标文件的详细信息对标（文件名、大小、最后修改时间、缩略图/图标）。
        - **决策按钮组**：提供 [覆盖]、[跳过]、[重命名]、[取消任务] 四个核心选项。
        - **策略记忆**：增加一个勾选项 **“为该任务中后续所有冲突应用此操作”**。勾选后，后端将自动应用该策略，不再弹窗干预。
    - **算法破解弹窗**：
        - 简易决策：告知用户已穷尽所有已知密码，询问是否启动深层算法破解（告知可能消耗较高的 CPU 资源）。
        - 按钮组：[授权启动] [放弃并标记失败]。
- **状态反馈**：任务在等待决策期间，进度条进入“琥珀色呼吸灯”效果，示意任务尚未结束但处于暂停。

### [FE-FLOW-002] 终态保留与错误审查 (State Retention)
- **目标**：满足“失败保留”的需求，提供可追溯的错误日志。
- **方案**：
    - **修改 `TaskStore.ts`**：在 `updateTaskStatus` 或接收到失败进度时，**绝不**将该任务从 `tasks` 数组中 `splice` 删除。
    - **UI 呈现**：失败的任务显示为带有红底色的卡片，显示错误原因（如：“密码均无效”或“文件已损坏”）。
    - **清理机制**：只有当用户手动点击“清除已完成/失败任务”按钮时，才将其从视图中移除。

## 5. 验收标准 (Acceptance Criteria)
1. **静默优先**：在设置有有效密码的场景下，解压过程100%无需用户点击任何弹窗，直接进入 `Completed` 状态。
2. **并发硬约束**：即使拖入 50 个文件，同时处理的（状态为 `Extracting` 或 `Pre-checking` 的）任务数绝不会超过 `config.max_concurrent_files`。
3. **挂起响应性**：故意制造一个同名冲突。观察到队列中该任务停滞并占用 Slot，后续任务不抢占；用户点击“跳过”后，任务立即完成并释放 Slot。
4. **失败可追溯**：设置一个不存在的密码。观察到任务在遍历完数据库和暴力字典后，稳稳地停留在 UI 列表中并标记为 `Failed`，用户可展开查看“所有已知密码均匹配失败”的日志。
